# Sprint 6 — первичные документы: заметки по безопасности

## Контроль доступа

- Все эндпоинты `/api/v1/primary-documents` завязаны на `get_current_user`; выборки и изменения фильтруются по `organization_id`. Обход по `doc_id` чужой организации даёт 404, не 403 — это снижает утечку факта существования идентификатора.

## Связи и валидация

- `counterparty_id` и `transaction_id` проверяются на принадлежность текущей организации.
- `related_document_id` должен ссылаться на документ той же организации; для типов `act` и `waybill` связь допускается только с документом типа `invoice`, чтобы не строить произвольный граф без бухгалтерского смысла.

## Печать PDF

- Текст в PDF проходит экранирование для ReportLab `Paragraph` (XML-спецсимволы), чтобы снизить риск неожиданной разметки в генераторе.
- Заголовок `Content-Disposition`: ASCII-имя для совместимости + `filename*` (RFC 5987) для кириллицы; без путей и без `..`.
- JSON-превью печати (`format=json`) доступно **только** при `DEBUG=True` в настройках; в проде по умолчанию отдаётся только `application/pdf`.

## Нумерация

- Счётчик хранится в `primary_document_sequences` с блокировкой строки `SELECT … FOR UPDATE` (на PostgreSQL); при высокой конкурентности возможны гонки на SQLite — тестовая/однопользовательская среда.

## Эксплуатация и миграции

- В Docker-образе API перед `uvicorn` выполняется `alembic upgrade head` (идемпотентные ревизии `sprint6_primary_docs`, `sprint7_onec_contours`): догоняют существующую PostgreSQL-схему без полного сброса БД.
- Новые таблицы также подхватываются при чистом `create_all` на пустой БД; при пустой схеме миграции no-op и только фиксируют ревизию.

## Рекомендации на будущее

- Ограничить частоту `GET …/print` (анти-DoS) при необходимости общим rate limit на API.
- Логировать выгрузку PDF в audit trail, если появится требование compliance.

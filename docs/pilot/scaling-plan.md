# План масштабирования ФинКлик
## 10 → 100 → 500 → 1000 клиентов

---

## Фаза 1: 10 → 100 клиентов (месяцы 1–3)

### Инфраструктура
- **Сервер:** 1 VPS (4 CPU, 8 GB RAM) — достаточно
- **БД:** PostgreSQL на том же сервере (до 50 кл.)
- **Redis:** одна нода 1 GB
- **Деплой:** Docker Compose
- **Стоимость:** ~50 USD/мес

### Что нужно сделать
- [ ] Открыть публичную регистрацию
- [ ] Настроить платёжный шлюз (ЕРИП / карты)
- [ ] Онбординг-автоматизация (welcome email, подсказки)
- [ ] Чат поддержки (Telegram бот)
- [ ] Мониторинг (Grafana + Prometheus)

### Метрики для перехода к Фазе 2
- 80+ активных платящих клиентов
- MRR > 4 600 BYN (~2 300 USD)
- Churn < 5%/мес

---

## Фаза 2: 100 → 500 клиентов (месяцы 3–9)

### Инфраструктура
- **БД:** Managed PostgreSQL (Yandex Cloud), 2 ноды
- **Redis:** Managed Redis, 2 GB
- **App:** 2–3 реплики API за балансировщиком
- **AI:** отдельный сервер для inference
- **CDN:** статика через Yandex CDN
- **Деплой:** Kubernetes (минимальный)
- **Стоимость:** ~300 USD/мес

### Что нужно сделать
- [ ] Перенос на Managed PostgreSQL
- [ ] Kubernetes деплой (через Helm)
- [ ] HPA: auto-scale API 2–10 реплик
- [ ] Разделить read/write трафик БД
- [ ] S3 для хранения документов
- [ ] Фоновые задачи (Celery/RQ для OCR)
- [ ] Партиционирование таблицы transactions по дате

### Метрики для перехода к Фазе 3
- 400+ активных клиентов
- MRR > 23 000 BYN (~11 500 USD)
- p95 latency < 300ms под нагрузкой
- Uptime > 99.5%

---

## Фаза 3: 500 → 1000 клиентов (месяцы 9–18)

### Инфраструктура
- **БД:** PostgreSQL cluster (master + 2 replica)
- **Cache:** Redis Cluster 3 ноды
- **App:** Kubernetes, 5–10 реплик, multi-zone
- **AI:** GPU inference (для ускорения OCR)
- **Queue:** RabbitMQ кластер
- **Стоимость:** ~1 000 USD/мес

### Что нужно сделать
- [ ] Multi-tenant изоляция данных (row-level security)
- [ ] Шардирование по org_id при >500 кл.
- [ ] Read replicas для аналитических запросов
- [ ] Message queue для всех async операций
- [ ] SOC 2 / ISO 27001 подготовка
- [ ] API rate limiting per-tenant
- [ ] Экспорт данных (GDPR-ready)

## Product Roadmap (Спринты 4-18)

Ниже — продуктовый трек развития ядра ФинКлик. Платформа независимая; white-label для банков остаётся отдельной услугой дистрибуции.

### Фаза 1 — Ядро учёта и 1С (спринты 4-7)

#### Спринт 4. Интеграция с 1С: подключение
- Протокол взаимодействия: HTTP-сервисы 1С (OData или кастомные API).
- Конфиг подключения на уровне организации (`onec_endpoint`, `onec_token`).
- Реальный health-check 1С-контура.
- Синхронизация справочника контрагентов (1С -> FinKlik).
- Импорт плана счетов из реальной базы 1С.

#### Спринт 5. 1С: синхронизация операций
- Двусторонняя синхронизация транзакций (FinKlik <-> 1С).
- Очередь задач для асинхронной синхронизации (Celery / ARQ).
- Обработка конфликтов и ошибок, retry, статусы синхронизации в UI.
- Логирование аудита синхронизации.

#### Спринт 6. Первичная документация (счета, акты, накладные)
- Модели: `Invoice`, `Act`, `Waybill` с привязкой к контрагенту и транзакции.
- Шаблоны печатных форм (PDF) под ИП и ООО.
- Нумерация документов по организации.
- CRUD + фронт: создание, список, печать/скачивание.
- Сквозной сценарий: выставить счёт -> получить оплату -> сформировать акт.

#### Спринт 7. 1С: провижининг контуров
- Автоматическое создание ИБ (или tenant в шарде) при регистрации клиента.
- Скрипты/API для массового обновления конфигурации.
- Маппинг `organization_id -> 1С контур` в реестре.
- Мониторинг здоровья контуров.

### Фаза 2 — Банк и платежи (спринты 8-9)

#### Спринт 8. Банковская интеграция
- Подключение к API банка (Беларусбанк / Альфа / Приор — по договору).
- Получение выписок, баланса, статусов платежей.
- Автоматический импорт банковских операций в транзакции FinKlik.
- Сверка банк <-> учёт.

#### Спринт 9. QR-оплата и платёжные ссылки
- Генерация QR-кода (ЕРИП или банковский протокол) из счёта.
- Отправка ссылки на оплату клиенту (email / мессенджер).
- Webhook / polling статуса оплаты с автоматической отметкой в учёте.
- Фронт: QR на странице счёта, статус `оплачен / ожидает`.

### Фаза 3 — Отчётность и порталы (спринты 10-12)

#### Спринт 10. Налоговый калькулятор — production
- Точные ставки и правила УСН / НДС по законодательству РБ.
- Расчёт взносов ФСЗН, Белгосстрах.
- Валидация: сверка с данными 1С.
- Фронт: детализация расчёта и подсказки.

#### Спринт 11. Формирование регламентированных отчётов
- Реальные форматы отчётов (ПУ-3, декларация УСН, НДС и др.).
- Заполнение из данных учёта (1С / FinKlik).
- Предпросмотр, редактирование, валидация перед отправкой.
- Версионирование форматов при смене нормативки.

#### Спринт 12. Подача отчётов на госпорталы
- Интеграция с порталами ФСЗН, МНС, Белгосстрах, Белстат.
- ЭЦП (если требуется): интеграция с криптопровайдером.
- Статусы: `подготовлен -> подтверждён -> отправлен -> принят/отклонён`.
- Уведомления о результатах и архив поданных отчётов.

**Статус в коде (2026-04):** экспорт отчётов, мок-подача и UI «Подача отчётов» частично закрывают эту фазу; нумерация фактических релизов в `docs/dev/` может не совпадать с номерами спринтов здесь — сводка: [`docs/dev/PRODUCT_SPRINT_ALIGNMENT.md`](../dev/PRODUCT_SPRINT_ALIGNMENT.md).

### Фаза 4 — ИИ, OCR, уведомления (спринты 13-15)

#### Спринт 13. Production OCR (сканер документов)
- Интеграция с OCR-сервисом (Tesseract / Google Vision / Яндекс).
- Распознавание ТТН, счетов-фактур, актов.
- Автоматическое создание транзакции/документа из скана.
- Фронт: фото с телефона -> результат -> подтверждение.

#### Спринт 14. ИИ-ассистент — production
- RAG: индексация регламентов, инструкций, FAQ по РБ.
- Контекст организации: сроки подачи, начисления, обязательства.
- Черновики документов с подтверждением пользователя.
- Ограничение: ИИ не меняет данные без явного действия.

#### Спринт 15. Уведомления и WebSocket
- Push-уведомления: дедлайны, статусы отчётов, оплаты.
- Email-дайджест (еженедельный/ежемесячный).
- Telegram-бот (опционально): напоминания.
- WebSocket: реальные события (новая оплата, статус отчёта).

### Фаза 5 — Масштаб и мобильность (спринты 16-18)

#### Спринт 16. PWA / мобильное приложение
- PWA manifest + service worker (offline-read режим).
- Адаптивный UI (доработка mobile UX).
- Альтернатива: React Native / Expo обёртка для App Store / Google Play.

#### Спринт 17. Нагрузочное тестирование и масштабирование
- Нагрузочные тесты (Locust / k6), целевая имитация 1000+ клиентов.
- Горизонтальное масштабирование API (несколько реплик).
- Шардирование 1С-контуров, балансировка клиентов по шардам.
- CDN для статики фронта.

#### Спринт 18. Безопасность и комплаенс
- Аудит безопасности (OWASP Top 10).
- Шифрование PII в базе до целевого стандарта.
- Резервное копирование и восстановление (автоматизация).
- Политика хранения данных и регуляторные требования РБ.
- Пентест.

### KPI и критерии готовности на 90 дней (контур спринтов 4-9)
- Успешность синхронизации 1С: >98%.
- Доля авто-сверки банк <-> учёт: >90%.
- p95 latency ключевых API: <400 ms.
- MTTR по sync-ошибкам: <30 минут.
- E2E-сценарий `счёт -> оплата -> акт`: 100% в smoke-тестах.

### Ключевые зависимости
- Стабильный health-check и конфиг 1С — до двустороннего sync.
- Типизированные документы — после стабилизации sync транзакций.
- Банковская сверка — до масштабирования QR/payment-link сценариев.
- Госпорталы и ЭЦП — после стабилизации генерации и валидации отчётов.

Единая финансово-учётная платформа для бизнеса Беларуси

Операции, банк, первичка, налоги, отчётность, кадры, контрагенты, ИИ и мобильный клиент — в одном контуре данных. Ниже раскрыт полный потенциал системы: как продукт выглядит для клиента и партнёра «end-to-end».

Навигация: стрелки / пробел / свайп. Материал для презентации инвесторам, банку white-label и enterprise-заказчику.

360°

От первички до статуса отчёта и push-уведомления — без разрыва между «учётом» и «действием».

Рынок и боль

Учёт разъезжается между Excel, банком, 1С и «головой бухгалтера»

  • Нет единого места, где видны деньги, налоги, сроки и документы одновременно.
  • Импорт выписок, сканы, первичка и отчётность живут в разных окнах — растёт риск ошибки и штрафов.
  • Владелец не понимает «что делать сегодня»: дедлайн, оплата, согласование, отправка в орган.
  • Банку и экосистеме не хватает «учётного слоя» поверх транзакций — чтобы удержать МСП в своём контуре.

ФинКлик закрывает разрыв

Один источник правды по операциям, документам, налогам и статусам подачи — с API и white-label для дистрибуции.

Кому это критично

ИП и ООО на УСН/НДС, с банковским оборотом, с кадрами и с регулярной отчётностью в ИМНС, ФСЗН, Белстат, Белгосстрах.

Позиционирование

ФинКлик — «операционная система» финансового compliance для МСП

  • Учёт: журнал операций, категории, импорт, сверка с банком, связка с 1С.
  • Документы: счета, акты, накладные, печать PDF, оплата по ссылке/QR, таймлайн событий оплаты.
  • Налоги и отчётность: калькуляторы, агрегаты ПУ-3, подготовка пакетов, подача в адаптер портала, WebSocket и архив снимка на момент отправки.
  • ИИ: консультант с RAG по официальным ориентирам, изолированный ключ организации (BYOK).
  • Мобильность: PWA, app-shell, safe-area, липкие действия в формах — готовность к нативной оболочке.

Модуль «Банк»

Мульти-банк, платежи, выписки и сверка с учётом

  • Подключение нескольких счетов, основной счёт, активность.
  • Импорт выписок (JSON/адаптеры) с дедупликацией и отражением в операциях.
  • Платёжные сценарии и сверка «книга ↔ импорт» за период — основа для банковского white-label.
  • Проверка контрагента по УНП через контур 1С — снижение мошенничества и ошибок реквизитов.

Для банка

Готовый «учётный слой» поверх API выписок: удержание клиента, кросс-продукт, снижение нагрузки на поддержку.

Для клиента

Меньше ручного ввода, больше автоматической классификации и прозрачности остатков.

Модуль «Отчётность»

От черновика до архивного снимка подачи

  • Подготовка отчётов по ведомствам, периодам и формам; предпросмотр и выгрузки в UI.
  • Подача через адаптер (mock / http), симуляция accept/reject в демо, возврат в черновик.
  • WebSocket report_status и уведомления — операционная осведомлённость без опроса статуса вручную.
  • Архивный снимок данных на момент отправки — аудит и разбор спорных ситуаций.

Архитектура и надёжность

Backend, миграции, тесты и готовность к продакшену

  • FastAPI + Alembic, асинхронная БД, модульная доменная логика.
  • CI: линт, тесты, сборка фронта, smoke-сценарии.
  • Безопасность паролей: PBKDF2 для новых пользователей, совместимость со старыми bcrypt-хешами.
  • Наблюдаемость: заготовки под Prometheus/Grafana в инфраструктурных файлах репозитория.

Мобильное приложение

Тот же продукт — в формате «app-first»

  • PWA manifest, иконка, standalone-режим; детект app-shell для компактного UI.
  • Safe-area, увеличенные tap-targets, отступы под нижнюю навигацию.
  • Липкие панели действий в модальных формах — как в нативных приложениях.
  • Следующий шаг инженерии: Capacitor/React Native shell без смены API.

Экосистема

Банк · 1С · Госорганы · ИИ — в одной истории клиента

  • Банк: выписки и платежи автоматически наполняют учёт.
  • 1С: синхронизация операций и справочников, очередь задач, retry.
  • Госорганы: подготовка и подача отчётности, статусы, архив.
  • ИИ: объясняет нормы, помогает сформулировать шаги, не подменяет юридическую ответственность.
  • Команда: роли, приглашения, тарифные лимиты — готово к организациям с несколькими пользователями.

Метрики успеха

Что измеряем при масштабировании

Время ввода

Снижение минут на операцию за счёт импорта, сканера и шаблонов первички.

Соблюдение сроков

Доля отчётов подготовленных до дедлайна; SLA реакции на reject.

Качество данных

Доля автоматически сопоставленных банковских строк; ошибки синка 1С.

Удержание

WAU/MAU, глубина сценария (сколько модулей использует организация).

Горизонт развития Vision

Уже сейчас — ядро; дальше — глубина интеграций и масштаб

  • Сейчас: полный UX ядра, демо-контуры госпорталов, OCR-пайплайн, отчётность и уведомления.
  • Ближайший этап: нативная оболочка, push, offline-read критичных экранов.
  • Средний этап: прямые API банков и порталов по договорённости, ЭЦП, очереди долгих задач.
  • Зрелость: мульти-тенант изоляция, расширенный аудит, нагрузочные профили под тысячи организаций.

Детализация отложенных внешних контуров зафиксирована в репозитории (sprint deferred) — для прозрачности с инвестором и регулятором.

Сценарная математика · 5 лет

Потенциал капитализации при зрелой экосистеме (Year 5)

В Year 5 добавляется эффект платформы: embedded finance, white-label в банках, партнёрские каналы 1С/интеграторов. При той же тарифной сетке 50/150/250 руб рост капитализации идёт за счёт объёма базы, расширения доли Pro/Business и снижения churn. Диапазон мультипликатора: ARR × 5–12.

10 000 клиентов

ARR: $2.4–7.2 млн
Возможная оценка: $12–86 млн
Реалистичный базовый коридор: $25–45 млн.

20 000 клиентов

ARR: $4.8–14.4 млн
Возможная оценка: $24–173 млн
Базовый коридор: $50–90 млн.

40 000+ клиентов

ARR: $9.6–28.8+ млн
Возможная оценка: $48–345+ млн
Upside при низком churn: $120–220+ млн.

  • Консервативный сценарий: низкий ARPA, медленный sales-cycle, ограничения по интеграциям.
  • Базовый сценарий: стабильный рост SMB + партнёрский канал через банк/интегратора.
  • Агрессивный сценарий: сильный white-label distribution и расширение продуктовой линейки.

Это сценарная оценка, не оферта. Для инвест-комитета обычно фиксируют 3 кейса: downside / base / upside с квартальным ревью факта против плана.
